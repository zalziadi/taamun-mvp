# Milestone v2.2 — Historical Observability

**Status:** ✅ Shipped
**Shipped:** 2026-04-20
**Commits:** 1 (v2.2 — 3 phases)

---

## Mission

v2.1 gave a live snapshot of ops metrics. v2.2 turns that snapshot
into a trend line — so the founder can see **change**, not just
state. Single migration, one daily cron, inline SVG sparklines, CSV
export.

---

## Shipped phases

### Phase 1: `ops_snapshots` migration + daily cron
- `supabase/migrations/20260421000000_ops_snapshots.sql` — tiny table with `UNIQUE(captured_on, metric)`, metric-first index, RLS on (service-role only)
- `src/lib/ops-metrics.ts` — `METRIC_SPECS[]` canonical list of 16 metrics, `headCount()` helper, `captureAllMetrics()` builder
- `src/app/api/cron/ops-snapshot/route.ts` — daily 23:55 UTC, upserts one row per metric so reruns are safe
- `vercel.json` — new cron entry

### Phase 2: Trend sparklines on `/admin/ops`
- `src/app/api/ops/trends/route.ts` — admin endpoint returning 30-day (configurable up to 365) series per metric
- `src/components/Sparkline.tsx` — pure-SVG sparkline, zero deps, auto-scaled, returns null for <2 points
- `/api/ops/stats` tiles now include `metricKey` so the dashboard can wire trends to tiles
- `/admin/ops` loads trends in parallel with stats; each tile gets an inline sparkline underneath when 2+ days of data exist

### Phase 3: CSV export
- `src/app/api/ops/export/route.ts` — streams full `ops_snapshots` table as CSV with proper filename + no-store cache
- "تصدير CSV" button in the `/admin/ops` header row alongside refresh

---

## Non-functional compliance

- Zero new dependencies
- Storage cost: ~6k rows/year at full 16-metric capture
- Sparklines degrade gracefully (hidden until data exists)
- All reads gated by `requireAdmin()`
- Cron uses upsert — manual reruns are idempotent
