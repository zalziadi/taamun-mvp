# Milestone v2.3 — Alert Thresholds

**Status:** ✅ Shipped
**Shipped:** 2026-04-20
**Commits:** 1 (v2.3 — 3 phases)

---

## Mission

v2.2 captured trends. v2.3 **pings the founder when a metric crosses
a line** — threshold-based alerts on top of `ops_snapshots`, delivered
via the existing Resend stack. Debounced 12h per rule.

---

## Shipped phases

### Phase 1: `ops_alerts` migration + CRUD API
- `supabase/migrations/20260421100000_ops_alerts.sql` — table with metric, comparison, threshold, label, active, last_fired_at
- Seeds 3 default rules on first install: threads_flagged > 0, journeys_flagged > 0, thread_replies_flagged > 0
- `src/app/api/ops/alerts/route.ts` — GET/POST/PATCH/DELETE, admin-gated, validates against `METRIC_SPECS` + comparison whitelist

### Phase 2: `/api/cron/check-alerts`
- Daily 00:05 UTC (right after ops-snapshot at 23:55)
- Fresh live `captureAllMetrics()` (doesn't rely on yesterday's snapshot)
- Evaluates every active rule; 12h per-rule debounce via `last_fired_at`
- Sends a single Arabic RTL digest email to `ADMIN_EMAIL` via Resend
- Silent no-op when email env missing or nothing triggered
- `vercel.json` — new cron entry

### Phase 3: `/admin/ops/alerts` CRUD UI
- Create form: metric dropdown (all 16 from METRIC_SPECS), comparison (5 operators), threshold input, optional label
- List view: active/inactive toggle, delete, last-fired timestamp
- Linked from `/admin/ops` header alongside "تصدير CSV" and refresh

---

## Non-functional compliance

- Zero new dependencies
- Reuses existing Resend integration (no new auth)
- All routes admin-gated via `requireAdmin()`
- Rule evaluator is a pure function (5 operators, no exec/eval)
- 12h debounce prevents email storms on persistent conditions
