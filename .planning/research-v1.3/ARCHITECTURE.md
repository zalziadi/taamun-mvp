## Feature 1 — Reflection Themes Clustering

### What exists today

- `src/app/api/cron/analyze-patterns/route.ts` — runs daily, writes one `pattern_insights` row per active user with:
  - `themes TEXT[]` (2-5 themes per day, AI-generated by Claude Haiku)
  - `weekly_summary TEXT` on `cycleDay % 7 === 0`
  - `district`, `awareness_state`, `depth_score`, `shift_detected` — already the richer feature vector.
- `pgvector` extension is enabled (`20260310000000_reflection_rag_analytics.sql`). Currently used only for `book_chunks`.
- `embedText()` in `src/lib/rag.ts` → OpenAI `text-embedding-3-small` (1536 dims).

### Recommendation: extend, don't replace

**Do NOT create `/api/cron/cluster-themes`.** Instead:

1. **New monthly cron route** `/api/cron/aggregate-themes` (separate from daily `analyze-patterns` — different cadence, different failure domain, different auth scope).
2. **New table `reflection_themes`** — shape below.
3. Clustering strategy: since `pattern_insights.themes` is already AI-distilled to 2-5 labels per day, the monthly job can **frequency-count + dedupe + merge-by-similarity**, which is O(n) in rows per user-month. This avoids in-memory k-means entirely.
4. If semantic dedupe is needed (e.g., "صبر" vs "الصبر" vs "تحمّل" should collapse): embed each unique theme label once via `embedText`, then cluster with a cosine-similarity threshold (~0.85) — a simple agglomerative pass, not k-means. Runs inside the cron handler, no DB-side clustering extension needed.

### Proposed schema

```sql
CREATE TABLE public.reflection_themes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  month_key TEXT NOT NULL,          -- e.g., "hijri-1447-09" or "greg-2026-04"
  theme_label TEXT NOT NULL,        -- canonical label (strongest member of cluster)
  theme_variants TEXT[] DEFAULT '{}', -- ["صبر","تحمّل","الصبر"]
  occurrence_count INT NOT NULL,    -- how many days this month it appeared
  confidence NUMERIC(3,2) NOT NULL, -- 0.00-1.00 (cluster cohesion)
  source_insight_ids UUID[] DEFAULT '{}', -- pattern_insights rows that fed this cluster
  generated_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, month_key, theme_label)
);

ALTER TABLE public.reflection_themes ENABLE ROW LEVEL SECURITY;
CREATE POLICY "own_themes" ON public.reflection_themes
  FOR SELECT USING (auth.uid() = user_id);
-- Writes: service_role only (cron)
```

**Why not `reflection_ids uuid[]`?** Because `pattern_insights` already aggregates reflections → themes. Pointing to `pattern_insights.id` (one per user-day) is a thinner fan-out.

### UX location

**Don't build a new `/themes` page.** Three lower-friction integration points:

1. **`/progress` page (existing)** — add a "أهم مواضيع شهرك" section below existing trajectory chart. Read-only. Primary surface.
2. **Monthly email digest** — `email_queue` infrastructure exists (`20260328000000_email_queue.sql`). Once-per-hijri-month push.
3. **Guide context** — inject top 3 monthly themes into `buildSystemPrompt` (`## مرحلة الرحلة` section already has `dominant_theme` from `user_journey`). Monthly themes give the guide richer priors.

### Cron schedule

- Run on hijri-month boundary (not greg). Use existing `src/lib/hijri.ts` (`getHijriDate()` already imported in `analyze-patterns`).
- Schedule: `vercel.json` cron at `0 2 * * *` daily, handler checks "is today hijri day 1?" and exits fast if not. Cheaper than dynamic schedule.

### Confidence: HIGH

## Feature 2 — Long-Term Memory for AI Guide

### What exists today

Reading the code contradicts the milestone brief's premise of a net-new `guide_memory` table:

- `guide_memory(user_id PK, soul_summary TEXT, themes JSONB, updated_at)` — shipped in `20260324000000_guide_sessions_memory.sql`.
- `generateSoulSummary()` in `src/lib/rag.ts` — Claude-generated cumulative prose summary, updated every 6 messages by `maybeUpdateSoulSummary()` in `src/app/api/guide/chat/route.ts`.
- `completeWithContext(question, contextChunks, systemPrompt, conversationHistory)` — context is injected as string array into the user message; history is passed separately. **No wrapper needed — `soul_summary` is already concatenated into `systemPrompt` by `buildSystemPrompt`'s "ذاكرة المشترك (soul)" section.**

### Two architectural paths

#### Path A (RECOMMENDED) — Deepen existing text summary

Keep the single-row `guide_memory` shape. Add:
- `key_moments JSONB` — array of `{date, insight_text, source_session_id}` capturing breakthrough moments detected by the guide itself (heuristic: `shift_detected = true` in `pattern_insights`, or user's own "عندي/أدركت/فهمت" utterances in session messages).
- `memory_priors JSONB` — small structured dict (recurring fears, named relationships, values mentioned ≥3 times) extracted by periodic summarizer.

**Why:** Current architecture is proven working end-to-end. Claude receives full `soul_summary` prose inside a system prompt — this is retrieval-augmented by design (Claude re-reads the summary every turn). Adding vectors is only justified if summary size exceeds prompt budget.

#### Path B — Add vector memory sibling

Only if Path A hits prompt-size limits, or if the guide is to surface specific past reflections ("قبل أسبوعين قلت...").

```sql
CREATE TABLE public.guide_memory_vectors (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  embedding VECTOR(1536) NOT NULL,
  summary_text TEXT NOT NULL,         -- one memory-sized chunk (~200-500 chars)
  source_type TEXT NOT NULL,          -- 'reflection' | 'guide_session' | 'insight'
  source_id UUID,                     -- foreign-key soft ref
  created_at TIMESTAMPTZ DEFAULT NOW(),
  last_used_at TIMESTAMPTZ
);

CREATE INDEX ON public.guide_memory_vectors
  USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);

ALTER TABLE public.guide_memory_vectors ENABLE ROW LEVEL SECURITY;
CREATE POLICY "own_memory_vectors" ON public.guide_memory_vectors
  FOR SELECT USING (auth.uid() = user_id);
```

**Retrieval:** Mirror `match_book_chunks` — new RPC `match_guide_memory(user_id, query_embedding, k)` with `WHERE user_id = $1` filter inside the function (per-user scoping is CRITICAL — do NOT rely on RLS for cross-user leak prevention inside an RPC called by service role).

**Integration point:** In `src/app/api/guide/chat/route.ts` after line 651 (the existing `match_book_chunks` block), add a parallel call:

```
const memoryContexts = await admin.rpc("match_guide_memory", {
  p_user_id: userId,
  query_embedding: embedding,
  match_count: 3,
});
// merge into ragContexts with a "ذاكرة شخصية:" prefix so completeWithContext distinguishes
```

**Recency weighting:** Do it in SQL via `ORDER BY (1 - (embedding <=> query)) * 0.7 + (recency_score) * 0.3`. Alternatively, post-sort in Node — cheaper for k=3.

### Write strategy

- **After every reflection save** — `reflections` table insert trigger NOT recommended (Supabase triggers + embeddings = latency). Instead: add embedding call to the reflection-save API route (which already exists) as fire-and-forget `void writeMemoryVector(...).catch(()=>{})`.
- **After every guide turn** — NOT every turn (expensive). Reuse the existing `maybeUpdateSoulSummary` cadence (every 6 messages) to also write one synthesized memory vector.
- **NEVER** fire embeddings synchronously in user-facing requests — always fire-and-forget, mirroring existing pattern.

### Privacy (NFR-04 carry-over)

- `guide_memory` and `guide_memory_vectors` are SACRED. PostHog already excludes sacred paths (NFR-04). Verify: `/api/guide/chat` and any new `/api/memory/*` routes must **not** emit any PostHog events, even failure events. Guard scripts `guard:phase-11` in package.json should add memory routes to the sacred list.
- Admin access: service-role writes only. No RLS bypass for debugging — use `SUPABASE_SERVICE_ROLE_KEY` in scripts, not admin dashboard.

### Recommendation

Ship **Path A first** (deepen text summary). Measure:
- (a) Does `soul_summary` regularly exceed ~1500 tokens? (prompt bloat)
- (b) Do users ask "do you remember when I said X?" style queries?

If either → add Path B as follow-up phase within v1.3.

### Confidence: HIGH on Path A recommendation, MEDIUM on Path B specifics (pgvector `ivfflat` lists=100 heuristic — validate against actual row count before shipping; for <10K rows, `hnsw` or flat scan is simpler).

## Feature 3 — Voice Journaling

### What exists today

**This feature is 80% shipped.** Files confirmed:

- `src/lib/munsit.ts` — REST client, `MUNSIT_API_KEY` env var, 2MB upload, webm/opus default.
- `src/app/api/voice/stt/route.ts` — Node runtime, `requireUser()` auth, 2MB cap, returns `{ok: true, text}`. **No audio storage** (Blob is consumed, never persisted). Matches milestone requirement exactly.
- `src/app/guide/voice/useVoiceSession.ts` — `MediaRecorder` hook, 60s max, idle/listening/thinking state machine, error taxonomy (`mic_denied | no_speech | stt_failed | chat_failed`). **Currently used by `/guide/voice` (voice chat with AI guide).**

### What's actually needed for v1.3

**Zero new backend code. Zero new infra.** Only:

1. **New client component** `src/components/VoiceInput.tsx` (smaller than `useVoiceSession` — this is dictation, not conversation):
   - Button: mic icon → hold-to-talk OR click-to-toggle.
   - On stop: POST audio blob to `/api/voice/stt`.
   - On `{ok: true, text}`: call `onTranscript(text)` callback.
   - Reuses the MediaRecorder plumbing from `useVoiceSession.ts` — extract shared primitive to `src/lib/voice/recorder.ts` rather than duplicating.

2. **Mount in `ReflectionJournal`** inside `DayExperience.tsx`. The textarea in `ReflectionJournal` should get a mic button in its corner. On transcript received → `setValue(prev => prev + transcript)` (append, don't replace — user may have typed already).

3. **A11y:** mic button needs `aria-label="ابدأ التسجيل الصوتي"` / `"إيقاف التسجيل"` (toggle), `aria-pressed={isRecording}`, and a visible recording indicator (not just icon color) for users who don't see red.

### Runtime constraint — already correct

`/api/voice/stt` uses `export const runtime = "nodejs"` — required because:
- Node's `FormData` streaming to Munsit API (Edge runtime has FormData but streaming uploads are brittle).
- 2MB payload → Edge's 4.5MB request body limit is fine, but Node is safer for multipart.

Keep as-is.

### Env var check

`MUNSIT_API_KEY` reference in `src/lib/munsit.ts` line 19 confirmed. Already referenced in `.planning/STATE.md`, `PROJECT.md`, and Munsit design doc. **Must exist in Vercel env.** Add to `docs/env.md` if not present.

### Privacy

Audio never touches disk: `req.formData()` → `Blob` → `fetch` to Munsit → response → return text → Blob garbage-collected. No audio in Supabase, no audio in logs (only `status` + `upstreamBody` on error — verify `upstreamBody` doesn't include audio metadata in Munsit error responses).

### Confidence: HIGH — all four components (lib, route, hook, env var) verified in repo.

## Phase Dependencies + Build Order

Given v1.3 starts at Phase 12 and continues v1.2's numbering, here is the recommended phase ordering (5 phases, possibly collapse to 4):

```
Phase 12: Voice Journaling integration  (3-5 days)
          └─ No new infra. UI + shared recorder primitive.
          └─ Ship first: user-visible win, no dependencies.

Phase 13: Long-Term Memory Path A  (4-7 days)
          └─ Depends on: nothing new.
          └─ Extends existing guide_memory + maybeUpdateSoulSummary.
          └─ Adds key_moments + memory_priors fields.
          └─ SHIPS EMBEDDING INFRASTRUCTURE (see note below).

Phase 14: Themes Clustering  (5-8 days)
          └─ Depends on: Phase 13's embedding helpers (shared with themes canonicalization).
          └─ New table + new monthly cron + /progress UI section.
          └─ Email digest deferred to follow-up if time-boxed.

Phase 15: A11y Audit  (3-5 days)
          └─ Depends on: Phase 12 + 13 + 14 shipped (so newly-added UI is audited too).
          └─ One doc, one remediation sweep, Lighthouse CI hook.

Phase 16 (optional): Long-Term Memory Path B  (5-8 days)
          └─ Only if Path A metrics justify it.
          └─ Depends on: Phase 13 embedding helpers (already shared).
```

### Critical dependency — EMBEDDING INFRASTRUCTURE

Themes (Feature 1) and Memory Path B (Feature 2) **both use `embedText()`** and **both write to pgvector tables**. The question asks "Where does memory come before themes?" — the answer is:

- **Themes comes AFTER memory** in the phase order because themes can use embeddings opportunistically (semantic dedupe of theme labels) but works fine without them (frequency-count-only). Memory Path B is the feature that *requires* embeddings and *requires* `ivfflat`/`hnsw` index tuning.
- **Shared primitives to extract in Phase 13:**
  - `src/lib/embeddings.ts` (new) — move `embedText` out of `rag.ts` (or re-export); add `embedBatch(strings: string[])` for batched theme label embedding.
  - `src/lib/pgvector-helpers.ts` (new) — canonical `match_*` RPC caller with per-user filtering, reusable by memory + themes.

### Why voice is Phase 12 (first)

- Zero backend dependencies.
- Most visible UX win (mic in the day experience).
- De-risks Munsit integration early in case env/quota issues surface.
- Unblocks manual user testing while memory + themes development proceeds in parallel branches.

### Why a11y is last

- Audits should cover the newly-added voice button, themes surface, and memory-influenced guide replies. Auditing pre-v1.3 and then again post-v1.3 doubles work.
- Single consolidated audit + remediation commit per component = cleaner history.
