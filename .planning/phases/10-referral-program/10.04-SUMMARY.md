## Ordering Rationale for the 5-Step Transaction

The FRIEND-* branch runs five mutations in a deliberate order. Reordering would break one of two invariants:

1. **Lookup → self-check** (pre-mutation guards) — zero DB writes if any guard fails.
2. **`profiles.upsert`** — this is the write that the invitee can actually feel ("I got a free month"). Done FIRST so a later failure never leaves them worse off.
3. **`profiles.update` (original_gateway tag)** — wrapped in try/catch; failure is logged, not blocking. Preserves Phase 9 renewal-CTA routing.
4. **`referrals.update`** — the audit record. If this fails AFTER step 2, we log and continue: the invitee already has their month; an admin can reconcile from `profiles.activated_at`. Rolling back step 2 would be strictly worse UX.
5. **`emitEvent` + cookie** — terminal; analytics is fire-and-forget (`emitEvent` never throws per `src/lib/analytics/server.ts` spec).

The alternative order (referrals.update before profiles.upsert) was rejected: if the upsert then failed, we'd have a `pending_day14` row pointing at a user with no subscription — a ghost reward for the referrer and no free month for the invitee.

## Decisions Made

- **FRIEND-* branch duplicates the 8-line cookie/return block** rather than extracting a shared helper. Factoring out a `buildActivationResponse(user, tier, expiresAt, via?)` would invent a signature before we know what a hypothetical third namespace (not in roadmap) would need — premature abstraction. Duplication is loud and auditable; both branches end identically.
- **App-layer self-referral reject (409) instead of letting DB CHECK fire**. The DB constraint (`referrals_no_self_referral`) would surface as a generic 500 from supabase-js; a typed 409 with `error: "self_referral_forbidden"` is friendlier and testable without a live DB. DB CHECK remains the backstop for any future code path.
- **Pass-through vitest alias mocks** for three `@/lib/*` modules (`entitlement`, `subscriptionDurations`, `referral/generate`). Required because vitest 4's default resolver doesn't honor the `@/*` tsconfig path alias. Adding a `vitest.config.ts` with the alias would be scope creep (new file + new convention for the project). `vi.mock(path, async () => await import(relativePath))` is one line per module and forwards to the real export — integration behavior (HMAC, calcExpiresAt, regex) is still exercised.
- **Kept the Phase 9 `original_gateway='eid_code'` tag**. A FRIEND-redeemed user is, from a billing-gateway perspective, first-gateway=`eid_code` (no Salla/Tap/Stripe charge). This keeps renewal-prompt routing consistent.
- **No `activation_codes` row minted for FRIEND-*** (explicit REFER-04 ban). The invitee reward is a direct `profiles.expires_at` update; the audit lives in `referrals`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Pass-through vi.mock() for `@/*` aliased modules**
- **Found during:** Task 1 (first test run)
- **Issue:** Vitest's default module resolver does not honor the `@/*` tsconfig path alias. The route under test imports 5 modules via `@/*`; 3 of them (`authz`, `supabaseAdmin`, `analytics/server`) were faked, but `entitlement`, `subscriptionDurations`, and later `referral/generate` were intended to run for real. Vitest raised `Cannot find package '@/lib/entitlement'`.
- **Fix:** Added three `vi.mock(path, async () => await import(relativePath))` pass-through factories. These intercept the alias before Node's resolver sees it and forward to the real module, preserving integration coverage (HMAC + calcExpiresAt + FRIEND_CODE_REGEX are the actual production code paths).
- **Alternative considered:** Adding `vitest.config.ts` with alias resolution — rejected as scope creep; no other test in the repo needs it (existing tests only reference `@/*` paths inside `vi.mock()` factories).
- **Files modified:** `src/app/api/activate/route.test.ts` only.
- **Verification:** 8/8 tests pass; the happy-path test asserts `expires_at ~ now+30d` and a valid HMAC'd cookie, proving the real `calcExpiresAt` + `makeEntitlementToken` ran.
- **Committed in:** `bda9a0e` (Task 1) and refined in `cef43d4` (Task 2 added the third mock for `referral/generate`).
