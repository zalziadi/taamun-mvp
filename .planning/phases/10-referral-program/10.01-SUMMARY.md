## Decisions Made
- **NEW table, not extension of activation_codes:** billing vs reward concerns separated per research §R1. REFER-02 locked this.
- **CHECK (invitee_id IS NULL OR referrer_id <> invitee_id):** NULL-tolerant so pending_invitee rows (invitee_id NULL) validate; app layer also enforces on redemption — DB is the backstop (REFER-07).
- **UNIQUE(referrer_id, invitee_id) with PG NULL-distinct semantics:** permits multiple pending rows per referrer pre-redemption, still blocks duplicate realized pairs.
- **Two SELECT policies vs one OR'd policy:** matches Phase 7 badges idiom; each policy independently assertable by verifier; clearer pg_policies output.
- **No INSERT/UPDATE/DELETE policies:** all writes through getSupabaseAdmin() (service role bypasses RLS). Mirrors existing badges + activation_codes pattern (REFER-12).
- **Partial index on status='pending_day14':** Plan 10.05 cron scans only this hot subset; partial index stays small as rewarded/voided rows accumulate.
- **ON DELETE CASCADE (referrer) + SET NULL (invitee):** data subject deletion symmetry for referrer; audit preservation for invitee-side deletions.
- **Status enum at DB layer:** locks the 5-state machine (pending_invitee, pending_day14, rewarded, refunded, void) so no downstream writer can silently introduce new states.
- **No new runtime dependencies** (CLAUDE.md rule #6 respected — SQL-only change, no new npm packages).
