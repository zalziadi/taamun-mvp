Required contents (in order):

1. Header comment block stating:
   - Phase 10 — v1.2 Retention Loop: Referrals table
   - REQs addressed: REFER-02, REFER-07, REFER-12
   - NFR-09 (additive only), NFR-08 (no new deps), NFR-10 (idempotent)
   - Scope boundary: does NOT seed any rows, does NOT touch activation_codes, does NOT write to profiles

2. `CREATE TABLE IF NOT EXISTS public.referrals` with columns:
   - `id uuid PRIMARY KEY DEFAULT gen_random_uuid()`
   - `code text NOT NULL UNIQUE` — the FRIEND-XXXX code value; UNIQUE so the same code cannot be handed out twice
   - `referrer_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE`
   - `invitee_id uuid NULL REFERENCES auth.users(id) ON DELETE SET NULL` — null until `/api/activate` redemption writes it
   - `status text NOT NULL DEFAULT 'pending_invitee' CHECK (status IN ('pending_invitee','pending_day14','rewarded','refunded','void'))` — per 10-CONTEXT §Storage state machine
   - `invitee_redeemed_at timestamptz NULL`
   - `referrer_rewarded_at timestamptz NULL`
   - `created_at timestamptz NOT NULL DEFAULT now()`
   - Table-level constraints:
     - `CONSTRAINT referrals_no_self_referral CHECK (invitee_id IS NULL OR referrer_id <> invitee_id)` — REFER-07 defense-in-depth (app layer also enforces; DB is the backstop)
     - `CONSTRAINT referrals_unique_pair UNIQUE (referrer_id, invitee_id)` — prevents same pair twice. Note: Postgres treats NULL != NULL in UNIQUE, so multiple pending_invitee rows per referrer (invitee_id NULL) are still allowed, which is what we want.

3. Indexes:
   - `CREATE INDEX IF NOT EXISTS idx_referrals_referrer_id ON public.referrals (referrer_id);` — /account/referral page reads
   - `CREATE INDEX IF NOT EXISTS idx_referrals_invitee_id ON public.referrals (invitee_id);` — /api/activate FRIEND-* lookup by invitee
   - `CREATE INDEX IF NOT EXISTS idx_referrals_status_redeemed ON public.referrals (status, invitee_redeemed_at) WHERE status = 'pending_day14';` — partial index for cron credit job (10.05) scan — keeps index tiny.

4. RLS (mirror the badges RLS pattern verbatim):
   - `ALTER TABLE public.referrals ENABLE ROW LEVEL SECURITY;`
   - `DO $$ BEGIN IF NOT EXISTS (...) THEN CREATE POLICY "users_select_own_referrals_as_referrer" ... FOR SELECT TO authenticated USING (auth.uid() = referrer_id); END IF; END $$;`
   - Second DO block for `users_select_own_referrals_as_invitee` with `USING (auth.uid() = invitee_id)` so invitees can see the referral row that credited them (transparent status per REFER-11).
   - NO INSERT/UPDATE/DELETE policies — all writes flow through `getSupabaseAdmin()` (service role), matching the /api/activate + badges existing pattern.

5. `COMMENT ON TABLE public.referrals IS 'v1.2 Phase 10 — Referral Program. FRIEND-XXXX codes + pending_day14 retention gate. Writes via service role only (RLS blocks client writes). See .planning/phases/10-referral-program/10-CONTEXT.md §Storage.';`

6. Commented-out DOWN block at the end (for operator reference only — not executed):
   ```
   -- DOWN (manual rollback):
   --   DROP TABLE IF EXISTS public.referrals CASCADE;
   ```
