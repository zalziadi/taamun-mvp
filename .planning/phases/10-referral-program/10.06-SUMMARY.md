## The 13 vitest cases

1. `isInviteeEligible` returns `'refunded'` when invitee `subscription_status='expired'` before redeemed+14d.
2. `isInviteeEligible` returns `'refunded'` when invitee `expires_at` < redeemed+14d (short month = refund).
3. `isInviteeEligible` returns `'day14_reached'` when `progress.completed_days` includes 14.
4. `isInviteeEligible` returns `'not_yet'` when max completed_day is 10.
5. `yearlyRewardedCount` filters by `referrer_id` + `status='rewarded'` + year-start ISO; returns the count.
6. `creditOneReferral` happy path → `'credited'` + `profiles.expires_at = max(now, current) + 30d` with correct formula.
7. `creditOneReferral` already-expired referrer → `newExpiresAt = now + 30d` (not `currentExpires + 30d`).
8. `creditOneReferral` refunded invitee → `'refunded'` + `referrals.status='refunded'` + **NO** `profiles.update` call.
9. `creditOneReferral` at-cap referrer (3 rewarded this year) → `'capped'` + `referrals.status='void'` + **NO** `profiles.update` call.
10. `creditOneReferral` not-yet-day14 → `'not_yet_day14'` + zero writes (row untouched, next run retries).
11. `creditOneReferral` idempotency: `row.status='rewarded'` → `'invalid_row'` + zero writes.
12. `creditOneReferral` idempotency: `row.status='void'` → `'invalid_row'` + zero writes.
13. `creditOneReferral` null `invitee_id` → `'invalid_row'` + zero writes (defensive guard).

## The exact `expires_at` math

```ts
const currentExpiresMs = referrer.expires_at ? Date.parse(referrer.expires_at) : 0;
const base = Math.max(Date.now(), Number.isFinite(currentExpiresMs) ? currentExpiresMs : 0);
const newExpiresAt = new Date(base + 30 * 24 * 60 * 60 * 1000).toISOString();
```

Rationale: always extend from the LATER of (current expiry, now). A referrer whose expiry is already in the past cannot stack past months — their reward starts from today. A referrer whose expiry is future gets a genuine 30-day extension.

## Decisions Made

- **newExpiresAt from `max(now, current)`** not from activation date. Rationale: `calcExpiresAt` in `subscriptionDurations.ts` computes from an activation date, which would either shorten already-future expiries or fail to stop past-month stacking. Inline `Math.max(...) + 30d` is the only formula that satisfies both defenses.
- **Refund heuristic NOT reading refund webhooks directly.** Rationale: Phase 10 scope doesn't include a dedicated refund webhook. `subscription_status='expired'` (set by the existing `manage-subscriptions` cron) + `expires_at < redeemed+14d` is a robust proxy: any refund that shortens the invitee's month will trip this.
- **Sequential credit loop (no `Promise.all`).** Rationale: each iteration re-reads `yearlyRewardedCount()` to catch cap races; parallelism would require SELECT FOR UPDATE semantics not available in the Supabase REST API. Sequential scales fine at v1.2 cap (max 3 rewarded per referrer per year → low pending_day14 density).
- **Comment-only mentions of `activation_codes` / `email_queue` / `emitEvent` preserved.** Rationale: the JSDoc explicitly documents the ban, making future editors aware. Per plan Task 1 action `action` list, the banned strings are not ALLOWED in executable code — grep guards validate this via pattern `from\("activation_codes"\)...insert` etc., which comments don't match.
- **Legacy `user_progress` table fallback kept.** Rationale: `src/lib/progressStore.ts` does the same fallback (code `42P01`); `readDay14Reached` mirrors the exact shape so cron-time day-14 truth aligns with app-time day-14 truth.
- **Idempotency via status-guard (not via `referrer_rewarded_at IS NULL`).** Rationale: a second run after partial failure (profiles.update succeeded, referrals.update failed) would still see `status='pending_day14'` and retry — cap check prevents runaway; `referrer_rewarded_at` is audit-only.
