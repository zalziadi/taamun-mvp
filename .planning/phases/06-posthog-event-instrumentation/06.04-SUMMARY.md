# Phase 6 Plan 04: day_complete Call Site Summary

Wire the FIRST production server-side event emission — `day_complete` fires from the `/api/program/progress` POST success handler with typed props `{day_number, cycle_number, tier}` — satisfying ANALYTICS-03 and establishing the end-to-end proof-of-pipeline that validates Plans 06.01 + 06.02.

## What Landed

### `src/app/api/program/progress/route.ts` — modified (+44 lines, -0 lines)

1. **New import** (line 8): `import { emitEvent } from "@/lib/analytics/server";` — server-only module, safe from Route Handlers.

2. **Tier fetch** (after `saved.ok` check): graceful-fallback single-column select against `profiles.subscription_tier`. Defaults to `"unknown"` on any error (missing profile row, schema gap, auth issue). Non-fatal.

3. **Cycle fetch** (after `saved.ok` check): graceful-fallback single-column select against `progress.current_cycle`. Defaults to `1` if the schema doesn't have the cycle columns yet (same pattern GET handler uses at lines 114-127).

4. **Fire-and-forget emission**:
   ```typescript
   void emitEvent(
     {
       name: "day_complete",
       properties: {
         day_number: day,
         cycle_number: Number(currentCycle ?? 1),
         tier,
       },
     },
     auth.user.id,
   );
   ```
   `void` prefix explicitly discards the promise — analytics must not add latency to the response. `emitEvent` already swallows errors internally (Plan 06.01 contract), so no try/catch is needed at the call site.

5. **Response shape preserved** — unchanged return payload `{ok, total_days, current_day, completed_days, completed_count, percent, day_completed}`. Zero breaking changes for the client.

## Control-Flow Guarantees

| Path | Emits event? | Why |
|---|---|---|
| Season closed (403) | NO | early return before emission |
| Unauthenticated (auth.response) | NO | early return |
| Invalid JSON body (400) | NO | early return |
| Invalid day (400) | NO | early return |
| `ensureUserProgress` fails (500) | NO | early return |
| `upsertUserProgress` returns `!saved.ok` (500) | NO | early return |
| `upsertUserProgress` succeeds | **YES** | emitEvent fires; structural placement after `if (!saved.ok) return ...` guarantees this |

Zero orphan telemetry on any failure path.

Canonical column for tier data in this project = `subscription_tier`. The event's typed `tier` property (a free-form `string` per the TypedEvent definition) holds the value `String(profileRow.subscription_tier)` — the Phase 6 contract is a property NAME of `tier`, not a column name of `tier`.

## Defense-in-Depth (All Plan 06.01 + 06.02 + 06.04 Gates Active)

A developer trying to leak PII through this specific call site would need to defeat ALL of:

1. **TypeScript narrowing** (06.01): TypedEvent union locks the `day_complete` variant to exactly `{day_number, cycle_number, tier}` — extra keys fail compilation.
2. **Runtime property guard** (06.02): `assertAllowedProperties()` runs inside `emitEvent()` BEFORE the fetch, rejecting any banned-pattern key even if the TypedEvent was widened via `as any`.
3. **Failure-path gating** (06.04 this): emission is structurally after `if (!saved.ok) return` — no orphan events.
4. **Fire-and-forget** (06.04 this): `void` discard means even an `emitEvent` crash in a future build cannot propagate into the response.

## Deviations from Plan

**1. [Rule 3 - Blocking] Plan critical-rule contradicted reality about `currentCycle` existence in POST**

- **Found during:** Task 1 code review, before editing
- **Issue:** The `<parallel_execution>` critical rule said "REUSE the POST handler's EXISTING `currentCycle` local variable — DO NOT add a second Supabase query for `progress.current_cycle`. The handler already computes it." But the actual POST handler (as of this commit's parent) does NOT compute `currentCycle` — that variable only exists in the GET handler (lines 115-127).
- **Fix:** Interpreted the rule by intent, not literal wording. The intent is "no duplicate queries within one handler." POST has ZERO existing cycle queries, so adding ONE is the first query, not a duplicate. I added the exact graceful-fallback pattern that GET uses (copy-paste shape), so the two handlers are now symmetric.
- **Files modified:** `src/app/api/program/progress/route.ts`
- **Commit:** `13c22b6`
- **Note for future plans:** If Phase 7 refactors progress state fetching into a shared helper (`getCurrentCycle(supabase, userId)`), both handlers can then call it — eliminating the current cross-handler duplication. Out of scope for 06.04.

**2. [Rule 2 - Missing critical functionality] Pre-step column verification surfaced canonical column name**

- **Found during:** Task 1 pre-step (column-verification grep)
- **Issue:** Plan example code used `profiles.tier`, but 15+ call sites in the codebase use `profiles.subscription_tier`. Writing `.select("tier")` would have returned `null` for every user (non-existent column), permanently defaulting `tier` to `"unknown"` in PostHog and making Phase 9 renewal funnel analysis impossible.
- **Fix:** Substituted `subscription_tier` for `tier` in the Supabase select, mapping to the event's `tier` property (event property name is still `tier` per the TypedEvent contract).
- **Files modified:** `src/app/api/program/progress/route.ts`

**3. Observability suggestions (posttooluse hook)** — The Vercel plugin suggested adding logging/error tracking to the route handler. Out of scope per CLAUDE.md S3 (minimal edits only); the existing GET handler already has no logging either. Deferred to any future observability pass.
