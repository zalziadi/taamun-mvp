# Phase 6 Plan 02: Runtime Property Whitelist Guard Summary

Add a runtime `assertAllowedProperties()` guard inside `emitEvent()` using a 7-regex banned-pattern list, so that even a future developer who widens a `TypedEvent` via `as any` cannot leak PII to PostHog — delivered as three surgical touches (1 new test file, 2 modified library files) with zero new dependencies and zero changes to downstream call sites.

## What Landed

### 1. `src/lib/analytics/events.ts` — extended (+62 lines)

Appended after the existing `ALLOWED_PROPERTY_KEYS` constant:

- **`BANNED_PROPERTY_PATTERNS: readonly RegExp[]`** — exactly 7 patterns, one per banned name-shape from 06-CONTEXT.md §"Property whitelist":

  | Pattern         | Regex              | Blocks examples                           |
  | --------------- | ------------------ | ----------------------------------------- |
  | `*_email`       | `/_email$/`        | `user_email`, `contact_email`, `something_email` |
  | `*_phone`       | `/_phone$/`        | `user_phone`, `primary_phone`             |
  | `reflection_*`  | `/^reflection_/`   | `reflection_text`, `reflection_length`    |
  | `verse_*`       | `/^verse_/`        | `verse_number`, `verse_text`              |
  | `journal_*`     | `/^journal_/`      | `journal_entry`, `journal_body`           |
  | `message_*`     | `/^message_/`      | `message_body`, `message_content`         |
  | `prayer_*`      | `/^prayer_/`       | `prayer_count`, `prayer_time`             |

- **`assertAllowedProperties(props: Record<string, unknown>): void`** — iterates keys × patterns; throws on first match with an error message naming both the offending key and the matching regex. O(n × 7) per call, n = number of properties (typically 2–4).

### 2. `src/lib/analytics/server.ts` — modified (+21 lines, -1 line)

- Upgraded the type-only `TypedEvent` import to also bring in `assertAllowedProperties`:
  ```typescript
  import { assertAllowedProperties, type TypedEvent } from "./events";
  ```
- Added a try/catch block as the FIRST statement inside `emitEvent()` — runs BEFORE the env-var check and BEFORE the fetch:
  - On throw in dev/CI (`NODE_ENV !== "production"`): error propagates so violations fail loudly.
  - On throw in prod: `console.warn` + early return. User-facing route response is never blocked; no network call is made with bad properties.

Existing behavior preserved:
- Silent no-op when `NEXT_PUBLIC_POSTHOG_KEY` / `NEXT_PUBLIC_POSTHOG_HOST` missing.
- Existing try/catch around `fetch` untouched — network errors still swallowed and logged.

### 3. `src/lib/analytics/events.test.ts` — new (108 lines)

Vitest suite with 14 test cases across 3 `describe` blocks:

| Group | Tests | Purpose |
|-------|-------|---------|
| Allowed property shapes | 4 | Positive cases: `day_number`, `badge_code`, empty bag, realistic `day_complete` payload |
| Banned property patterns (ANALYTICS-12) | 9 | One per banned pattern + `something_email` (glob variant) + mixed-keys case |
| `BANNED_PROPERTY_PATTERNS` export | 2 | Length === 7, all entries are `RegExp` instances |

Each banned-pattern test asserts the error message contains BOTH the offending key name AND the word "banned" (regex `/KEY.*banned/`), so message drift is detected.

## The 7 Banned Patterns (ANALYTICS-12)

```typescript
export const BANNED_PROPERTY_PATTERNS: readonly RegExp[] = [
  /_email$/,        // *_email
  /_phone$/,        // *_phone
  /^reflection_/,   // reflection_*
  /^verse_/,        // verse_*
  /^journal_/,      // journal_*
  /^message_/,      // message_*
  /^prayer_/,       // prayer_*
] as const;
```

Exact parity with 06-CONTEXT.md §"Property whitelist" line 55. When Plan 06.06 lands, the CI grep rule will re-derive its banned strings from this same source so drift is impossible.

## Defense-in-Depth

Three independent gates now protect against PII leakage to PostHog:

| Gate | Where | Owner | Bypassable by |
|------|-------|-------|---------------|
| TypeScript narrowing (`TypedEvent` discriminated union) | Compile time | Plan 06.01 | `as any` |
| Runtime regex guard (`assertAllowedProperties`) | Request time | **Plan 06.02 (this)** | Deleting the call — visible in diff |
| CI grep lint rule | Build time | Plan 06.06 (parallel) | Bypassing CI — visible in commit |

A developer would have to defeat all three to leak a banned property — and each bypass is individually visible in code review.
