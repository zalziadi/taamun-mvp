# Phase 6 Plan 01: TypedEvent Catalog + Server Emitter Summary

Establish the typed event contract and a server-side `emitEvent()` helper that every downstream v1.2 phase (06.02 through Phase 11) will import — delivered as two new files with zero new dependencies, zero modifications to the existing client-only `src/lib/analytics.ts`, and full preservation of the `person_profiles: "never"` privacy anchor.

## What Landed

### 1. `src/lib/analytics/events.ts` (new, 117 lines)

- Exports `TypedEvent` as a discriminated union with exactly 8 variants:
  - `day_complete` (Plan 06.04 owns call site)
  - `cycle_start` (Phase 7)
  - `badge_unlock` (Phase 8)
  - `renewal_prompted` (Phase 9)
  - `referral_code_generated` (Phase 10)
  - `referral_code_redeemed` (Phase 10)
  - `year_review_opened` (Phase 11)
  - `year_review_shared` (Phase 11)
- Exports `ALLOWED_PROPERTY_KEYS: readonly string[]` — 12 keys (day_number, cycle_number, new_cycle_number, prior_cycle_days_completed, milestone_code, badge_code, referral_code_prefix, renewal_days_remaining, gateway, tier, year_key, reflections_count).
- Zero imports from `posthog-js` or any client-only module — safe to import from Route Handlers and Server Components.
- Zero import of `../analytics` — avoids the `"use client"` transitive boundary.
- File-level JSDoc documents call-site ownership across Phases 6–11 and the banned-property-name contract from ANALYTICS-12.

### 2. `src/lib/analytics/server.ts` (new, 70 lines)

- Exports `emitEvent(event: TypedEvent, distinctId: string): Promise<void>`.
- Transport: native `fetch` POST to `${NEXT_PUBLIC_POSTHOG_HOST}/capture/` with body `{ api_key, event, distinct_id, properties, timestamp }` per PostHog's documented Capture API.
- Silent no-op when `NEXT_PUBLIC_POSTHOG_KEY` or `NEXT_PUBLIC_POSTHOG_HOST` missing (matches existing `initAnalytics()` pattern).
- Wraps `fetch` in try/catch; logs failures via `console.warn`; never throws, never rejects. Analytics failures cannot block a user-facing API route's success response.
- Does NOT touch `window` / `document` / `localStorage` — safe for Node.js runtime Route Handlers.

## Why Option B (New File) Instead of Option A (Append to analytics.ts)

The plan flagged both options but pre-decided Option B based on line 1 of `src/lib/analytics.ts`:

```typescript
"use client";  // line 1 — required by posthog-js
```

Next.js 14 App Router treats `"use client"` as a module-boundary declaration. Appending `emitEvent` to that file would mean:

- **Option A-a (keep "use client"):** Server Route Handlers importing `emitEvent` would also transitively pull in `posthog-js` initialization code, inflating Route Handler bundles and risking SSR-time references to `window`.
- **Option A-b (remove "use client"):** `posthog-js` would fail in the client because `initAnalytics()` directly references `window.localStorage` and `posthog.init()`, which Next.js would refuse to evaluate during SSR.

Neither is acceptable. Option B cleanly separates the two runtimes:

- `src/lib/analytics.ts` — client-only, unchanged, `"use client"` preserved.
- `src/lib/analytics/server.ts` — server-only, no pragma, fetch-based.
- `src/lib/analytics/events.ts` — shared pure-type module, usable from either side.
