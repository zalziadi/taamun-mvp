# Phase 6 Plan 06: CI Privacy Guard Summary

Build-breaking pure-Node grep guard that makes it impossible to ship a tracking pixel on a sacred page or a PII-shaped event property. Wired into `npm run guard:release` so the release gate fails before `tsc` and `build`.

## What Shipped

### 1. `scripts/guards/analytics-privacy.js` (292 lines, 0 new deps)

- **R1 (ANALYTICS-09)** — Line-based regex scan. Fails on `track(` or `posthog.capture(` in any file under:
  `src/app/day/**`, `src/app/reflection/**`, `src/app/book/**`, `src/app/program/day/**`, `src/app/api/guide/**`, `src/app/guide/**`
- **R2 (ANALYTICS-10)** — Line-based regex scan. Fails on `track(` or `posthog.capture(` inside any file whose basename is one of:
  `DayExperience.tsx`, `ReflectionJournal.tsx`, `AwarenessMeter.tsx`, `BookQuote.tsx`, `VerseBlock.tsx`, `HiddenLayer.tsx`, `SilenceGate.tsx`
- **R3 (ANALYTICS-12)** — Call-site-scoped scan. Walks each file's full source, finds every `emitEvent(`/`track(`/`posthog.capture(` occurrence, tracks proper paren depth (skipping string literals + comments), extracts the argument substring, then flags any of the banned property patterns inside it:
  `*_email`, `*_phone`, `reflection_*`, `verse_*`, `journal_*`, `message_*`, `prayer_*`
- **Exemptions:** the guard script itself is exempt from all rules. `src/lib/analytics/events.ts` and `src/lib/analytics/events.test.ts` are exempt from R3 (they define the patterns as literals; the plan pre-declares this so 06.01/06.02 don't trip the guard).
- **Output:** clean state prints `✓ analytics privacy guard passed (N files scanned)` and exits 0. Violations print `VIOLATION [REQ-ID] file:line \n  reason` per finding and exit 1.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] R3 regex scoping over-matched on first implementation**
- **Found during:** Task 1 first test run
- **Issue:** The plan's example code applied `BANNED_PROPERTY_REGEX` to every line of every file. On the current 438-file codebase this produced 12 false positives — all of them legitimate non-analytics code: Supabase `.update({ used_email })` in `/api/activate/route.ts`, `.insert({ verse_ref, message_count })` in admin cycle gen + guide chat, Google Vision client config `{ client_email }` in `/api/scan/route.ts`, Twitter DM API `{ message_create, message_data }` in an agent lib.
- **Fix:** Replaced the naive per-line regex with `extractEmitCallArgs(src)` — a full-source walker that:
  1. Finds every `emitEvent(`/`track(`/`posthog.capture(` opener.
  2. Tracks paren depth properly, skipping single/double/backtick string literals, line comments, block comments, and template-literal `${...}` expressions.
  3. Returns the exact argument substring for each call.
  4. Only then runs `BANNED_PROPERTY_REGEX` on those substrings.
- **Result:** Zero false positives. R3 still fires on real deliberate violations (Test 4 above proves it). Covered by the plan's `behavior` block point 3 which explicitly says "inside `emitEvent(` / `track(` / `posthog.capture(` call sites" — the plan's illustrative regex code was simpler than the stated behavior; this implementation matches the stated behavior.
