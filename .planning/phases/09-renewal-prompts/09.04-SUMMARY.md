# Phase 9 Plan 4: RenewalBanner + /api/renewal/status Summary

**Ships the user-facing half of Phase 9: a single client-side banner mounted in AppChrome that renders only within the 7-day expiry window, routes its CTA per the user's original payment gateway, dismisses for 48h with a 3-strike cap, fires `renewal_prompted` exactly once per session, and structurally refuses to render on any sacred path (`/day/**`, `/reflection/**`, `/book/**`, `/program/day/**`, `/guide/**`).**

## Accomplishments

- **Task 1 — `GET /api/renewal/status`**: auth-gated JSON endpoint. Calls `shouldShowRenewalBanner(userId)` from Plan 09.05, then invokes `refreshEntitlementIfStale(req, res, userId)` from Plan 09.06 so the response carries both the banner decision AND (when applicable) a re-minted entitlement cookie. `force-dynamic`, `nodejs` runtime. No PII. No analytics emission.
- **Task 2 — `<RenewalBanner />`**: client component. Sacred-path guard runs BEFORE any fetch. Status-fetch fires once in `useEffect` (no polling, no `setInterval`). LocalStorage keys: `taamun.renewal_dismissed_until.v1` (+48h on dismiss) and `taamun.renewal_dismiss_count.v1` (increments; cap 3). SessionStorage key `taamun.renewal_prompted_session.v1` gates analytics to one emit per session. Helpers split into `renewalBannerHelpers.ts` with 14 vitest cases.
- **Task 3 — Mount in AppChrome**: single import + three-line render block inside `<main>`, gated by `!hide` so chrome-less routes (/breathing, /journal, /tasbeeh, /ramadan/*, /admin/*) never carry the banner. The banner itself also checks `isExcludedPath`, giving two independent layers.

## Decisions Made

1. **Logic-only tests instead of RTL render tests.** The plan's behavior bullets asked for 7 React-Testing-Library scenarios (sacred-path render null, dismiss click, analytics once). RTL, jest-dom, and jsdom are not in `devDependencies`, and adding them violates CLAUDE.md rule 6 + Plan 09.04's own NFR-08. Covered the behavior at the pure-function layer (14 tests) and leaned on Plan 09.07's integration harness for end-to-end render verification. Documented as deviation #1 below.

2. **Helpers in `renewalBannerHelpers.ts` (not inline in the .tsx).** This repo has no vitest JSX transform (confirmed by Plan 09.05/09.06 summaries and by the failing initial test run). Splitting lets vitest import the pure logic directly. The component re-imports its own storage key constants from the helpers module, keeping a single source of truth.

3. **Refresh cookie AFTER shouldShow, not before.** Plan 09.06's canonical signature is `(req, res, userId)`; it attaches Set-Cookie onto the response. We build the response with the banner payload first, then hand it to the refresher — the outgoing cookie and body are on the same `NextResponse` instance so no cookie-propagation gymnastics are needed. This diverges from the plan's snippet (which used a placeholder response and manually propagated cookies), but the outcome is identical and the code is simpler.

4. **`renewal_prompted` skipped for `gateway==='eid_code'`.** The TypedEvent gateway union in `src/lib/analytics/events.ts` is `'salla' | 'tap' | 'stripe'` — eid_code users can't be typed into that event. Skip path still sets the sessionStorage flag so the "at most once per session" invariant is preserved even on repeat visits. Documented inline in the component.

5. **No framer-motion, no icon library.** Dismiss button uses the Unicode `✕` glyph, not `lucide-react` or any other icon import. Fade uses Tailwind `transition-opacity duration-300` only. Keeps the component under `NFR-08` and clears Plan 09.07's anti-pattern guard trivially.

6. **No inline day count in copy.** Even though the banner receives `daysRemaining` from the API, the rendered Arabic copy is static ("واصل معنا — اشتراكك يقترب من الانتهاء.") — no countdown digits appear in the UI. `daysRemaining` travels with the analytics event only. This is deliberate defense against Plan 09.07's scarcity-copy guard and against copy-by-accident drift.
