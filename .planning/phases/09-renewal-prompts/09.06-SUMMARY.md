# Phase 09 Plan 06: refreshEntitlementIfStale Summary

One-liner: **cookie-vs-DB reconciliation helper that re-mints the HMAC entitlement cookie when Supabase shows a later `profiles.expires_at` than the cookie currently carries — defending against Pitfall #14 (auto-renewed user still sees renewal banner because cookie hadn't caught up).**

## What Shipped

`src/lib/renewal/refreshEntitlementIfStale(req, res, userId): Promise<void>`:

1. Reads `taamun_entitled` cookie from `req.cookies`. No cookie → no-op.
2. Runs `verifyEntitlementToken`. Invalid/forged/legacy → no-op.
3. Queries `profiles.expires_at, subscription_tier` via supabase admin (read-only, `maybeSingle`).
4. If DB row missing, `expires_at` null, or supabase errors → no-op.
5. If `dbExpiryMs <= cookieExpiryMs` → no-op (equal = no refresh needed).
6. Otherwise mints a fresh token via `makeEntitlementToken(userId, tier, profile.expires_at)` and attaches it via `res.cookies.set(...)` with the same cookie options as `/api/activate/route.ts` (`httpOnly`, `sameSite: "lax"`, `path: "/"`, `secure` only in production, `maxAge` from `cookieMaxAge(tier)`).
7. Wraps the whole body in try/catch → `console.warn` and return. The helper NEVER throws.

## Tests (5/5 passing under vitest 4.1.2)

| # | Scenario                                               | Assertion                                           |
|---|--------------------------------------------------------|-----------------------------------------------------|
| 1 | DB expiry later than cookie (30d vs 2d)                | `res.cookies.set` called once with correct options  |
| 2 | DB expiry earlier than cookie (5d vs 10d)              | `res.cookies.set` NOT called                        |
| 3 | No entitlement cookie on request                       | No DB query, no `res.cookies.set`                   |
| 4 | Forged/invalid cookie (`verifyEntitlementToken` fails) | No DB query, no `res.cookies.set`                   |
| 5 | Supabase client throws                                 | Resolves without rejection, no `res.cookies.set`    |

## Key Decisions

1. **Canonical signature `(req, res, userId)`.** Plan 09.04's earlier illustrative snippet shows `(userId, res)` — that must be updated when 09.04 executes. `req` is required to read the current cookie (we can't mint a new one without knowing whether the old one even exists and what tier it carried for the `verified.tier` fallback).

2. **Strict inequality, no wiggle room.** `if (dbExpiryMs <= cookieExpiryMs) return`. Equal expiries are treated as "already in sync" — no wasted Set-Cookie on every request.

3. **Imports are relative (`../entitlement`), not `@/lib/entitlement`.** This project has no `vitest.config.*` file, so vitest does not resolve the TypeScript `@/*` path alias. The 10 existing passing test files in this repo (`src/lib/nextStep.test.ts` etc.) all use relative imports — we match that convention. Plan 09.05's sibling `shouldShow.ts` uses `@/` imports; its test file may need the same treatment when 09.05 runs.

4. **Tier fallback chain.** `profile.subscription_tier || verified.tier || "monthly"`. Protects against legacy profiles rows with null tier and keeps `cookieMaxAge(tier)` always producing a sane number.

## Deviations from Plan

None with respect to scope. The plan's action snippet was followed essentially verbatim; the only mechanical change was switching the two `@/lib/...` imports in the helper to relative paths so vitest can resolve them in this repo's setup (see Decision #3).

## Deferred Issues

**Pre-existing bug in `src/lib/entitlement.ts` (OUT OF SCOPE for Plan 09.06).**

- **What:** `verifyEntitlementToken` splits the base64-decoded token on `":"` to extract `[userId, tier, timestamp, expiresAt, signature]`. When `expiresAt` is a full ISO timestamp like `"2026-04-20T14:57:32.000Z"`, the embedded colons cause the split to produce 7+ parts. `parts[3]` ends up as `"2026-04-20T14"` (truncated), the signature-payload recomputation uses this truncated value, and the HMAC comparison fails — the token never round-trips as `valid:true` at all when expiresAt contains colons.
- **Impact on production:** `/api/activate/route.ts` calls `makeEntitlementToken(userId, tier, expiresAt)` with an ISO string from `calcExpiresAt`. Per this analysis, no such cookie would subsequently verify. This suggests either (a) production code paths tolerate `{valid: false}` via separate DB-truth checks, or (b) the bug lives in a code path that rarely executes. Either way, resolving it is out of scope for a cookie-vs-DB reconciliation plan.
- **Workaround in our tests:** We mint test tokens with colon-free date expiries (`"YYYY-MM-DD"` via `.toISOString().slice(0, 10)`), which round-trip cleanly. This keeps Plan 09.06's tests deterministic without patching `entitlement.ts`.
- **Recommended follow-up:** File a separate bug-fix plan that either (a) URL-encodes the `expiresAt` segment before embedding in the token, or (b) uses a non-`:` delimiter (e.g., `|`), or (c) serialises expiresAt as an epoch-ms number to avoid colons entirely. All three require a coordinated client-side cookie invalidation or a versioned token format.

## How To Use (from Plan 09.04's route handler)

```ts
// src/app/api/renewal/status/route.ts
import { refreshEntitlementIfStale } from "@/lib/renewal/refreshEntitlement";

export async function GET(req: NextRequest) {
  const auth = await requireUser();
  if (!auth.ok) return auth.response;

  const res = NextResponse.json(await buildPayload(auth.user.id));
  // Reconcile cookie BEFORE the client reads banner state — so the next
  // request reflects the freshest expiry.
  await refreshEntitlementIfStale(req, res, auth.user.id);
  return res;
}
```
