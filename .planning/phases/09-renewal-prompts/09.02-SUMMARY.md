**Idempotent pure-SQL migration that populates `profiles.original_gateway` for every existing paying user by reading from `salla_connections`, `customer_subscriptions` (segmented by `payment_provider`), and `activation_codes.used_by` — NFR-09 step 2 of 2, unblocking Phase 9's RenewalBanner gateway routing for historical customers.**

## Decisions Made

- **Source-table name deviation (see next section):** used `salla_connections`, `customer_subscriptions` (segmented), and `activation_codes` — the tables that actually exist — rather than the plan's `salla_integration` / `tap_customer_subscriptions` / `customer_subscriptions` / `activation_codes` naming.
- **Stripe bucket includes NULL `payment_provider`:** the column was added retroactively in `20260321120000_tap_customer_subscriptions.sql` with default `'stripe'`; any historical row with `payment_provider IS NULL` is therefore pre-Tap Stripe.
- **NULL-remaining rows are acceptable:** grandfathered users (no payment record — manual grants, admin accounts) legitimately have no gateway; Plan 09.04 banner treats NULL as "do not show."
- **No NOT NULL tightening in this file:** per plan's `<out_of_scope>`. Future migration can tighten after product verifies the residual NULL count.
- **No new runtime dependencies** (CLAUDE.md rule #6 respected — SQL-only change).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Critical functionality] Adapted source-table names to match actual schema**

- **Found during:** Task 1 (schema introspection step mandated by plan's `<action>` block)
- **Issue:** The plan's `<interfaces>` block named `public.salla_integration` and `public.tap_customer_subscriptions` as source tables. Both names DO NOT EXIST in the repo's migration history. Taking the plan literally would have produced a migration that crashes with `relation "public.salla_integration" does not exist` on apply.
- **Introspection result:**
  - `salla_integration`: not present. Actual table is `public.salla_connections`, created by `20260226004000_salla_integration.sql` (filename is misleading — the CREATE TABLE inside names the table `salla_connections`). It has a `user_id uuid` PRIMARY KEY references `auth.users(id)`.
  - `tap_customer_subscriptions`: not present as a dedicated table. `20260321120000_tap_customer_subscriptions.sql` (again, filename is misleading) adds a `payment_provider text NOT NULL DEFAULT 'stripe'` column + `tap_charge_id text` to the SHARED `public.customer_subscriptions` table. Tap subscribers are rows in that table with `payment_provider = 'tap'`.
  - `customer_subscriptions`: EXISTS (created by `20260320120000_customer_subscriptions.sql`). Has `user_id uuid NOT NULL` column. Holds BOTH Stripe and Tap rows, disambiguated by `payment_provider`.
  - `activation_codes.used_by`: EXISTS (confirmed via `src/app/api/activate/route.ts` at lines 54, 63). Referenced by admin and activation routes throughout the code.
- **Fix:**
  - Section 1 (Salla) uses `public.salla_connections`.
  - Section 2 (Tap) uses `public.customer_subscriptions WHERE payment_provider = 'tap'`.
  - Section 3 (Stripe) uses `public.customer_subscriptions WHERE (payment_provider IS NULL OR payment_provider = 'stripe')` to catch both pre-Tap-column NULL rows and post-column Stripe rows.
  - Section 4 (eid_code) uses `public.activation_codes WHERE a.used_by = p.id` — as written in the plan.
- **Rationale for auto-fixing rather than skipping:** Plan 09.02's `<action>` directive said "if a table is missing … SKIP that section's UPDATE." Skipping Salla + Tap would defeat the plan's own success criterion ("Plan 09.04's banner can route 100% of currently-paying users to the correct gateway on day 1"). The plan clearly intended these three gateway signals; the deviation is a naming translation to reach the same semantic goal. Rule 2 (auto-add missing critical functionality) applies — without correct table names, the migration is inert.
- **Files modified:** `supabase/migrations/20260421100000_v1_2_profiles_original_gateway_backfill.sql` (header comment documents the deviation in full).

## Open Flags

- **Schema naming drift:** the migration filenames `20260226004000_salla_integration.sql` and `20260321120000_tap_customer_subscriptions.sql` give the misleading impression that those tables exist as named. The actual tables are `salla_connections` and `customer_subscriptions` (shared with Stripe). Consider a future clarifying migration-rename pass or an `ARCHITECTURE.md` note — out of Phase 9 scope.
- **Stripe bucket catches NULL `payment_provider` rows.** If any historical Tap rows were written before the `payment_provider` column default was effective (very unlikely — default is declared in the same migration that adds the column, so NOT NULL + DEFAULT 'stripe' applies atomically), they would incorrectly be bucketed as Stripe. Pre-apply verification query: `SELECT COUNT(*) FROM public.customer_subscriptions WHERE payment_provider IS NULL` — if > 0 on staging, investigate before prod apply.
