# Phase 7 Plan 04: Day-28 Milestone Badge — Silent Reveal Summary

**One-liner:** Day-28 badge delivered as a silent side-effect of the cycle transition: a geometric SVG component + an idempotent `unlockBadge` helper (single source of truth for the exactly-once `badge_unlock` event) + a thin POST route + one-line wire into Plan 07.02's start-cycle success path — no modal, no toast, no share, no animation.

## Accomplishments

- **Task 1 — `src/components/badges/MilestoneBadge.tsx`:** Pure presentational SVG component. Day-28 variant only. Concentric gold-stroke circles + 4 cardinal ticks. Arabic name (`عتبة الثامن والعشرين`) and Eastern-Arabic numeral (`٢٨`) rendered as HTML sibling spans — SVG contains zero `<text>` nodes, satisfying PITFALLS.md #8 structural invariant. `unlocked=false` applies `opacity-40 grayscale`. No hooks, no effects, no interactive handlers, no share action.

- **Task 2 — helper extraction (`src/lib/badges/unlock.ts`) + route wrapper (`src/app/api/badges/unlock/route.ts`):**
  - Helper exports `unlockBadge(userId, "day_28", cycle_number, day_number): Promise<{ unlocked: boolean; reason?: "already_unlocked" | "db_error" }>`.
  - Uses service-role Supabase client + `.upsert(..., { onConflict: "user_id,badge_code,cycle_number", ignoreDuplicates: true }).select()`.
  - `.select()` returns the inserted row on first call (length 1) and empty array on duplicate (length 0); `inserted = data.length > 0`.
  - `emitEvent({ name: "badge_unlock", properties: { badge_code, day_number, cycle_number } }, userId)` fires **only** when `inserted === true`. Duplicate calls emit zero events.
  - Route is a ~60-line wrapper: auth via `createSupabaseServerClient`, validate `{ badge_code, cycle_number, day_number }` body, enforce `PHASE_7_ALLOWED_BADGE_CODES = {"day_28"}` scope guard (returns 400 with `{ error: "badge_code_not_in_phase_7_scope", allowed: ["day_28"] }`), delegates to `unlockBadge`.

- **Task 3 — wire into start-cycle (`src/app/api/program/start-cycle/route.ts`):** Added `import { unlockBadge } from "@/lib/badges/unlock";` and, after the progress update succeeds and **before** the existing `emitEvent("cycle_start", ...)` call, invoked `await unlockBadge(auth.user.id, "day_28", finishedCycle, 28)` inside a try/catch that is purely defensive (helper is documented as never-throwing). `finishedCycle` is `targetCycle - 1` — the badge belongs to the cycle the user just completed. `emitEvent` count in start-cycle remains exactly 1 (the `cycle_start` event); `badge_unlock` lives in the helper.
