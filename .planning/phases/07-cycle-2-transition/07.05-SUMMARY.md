# Phase 07 Plan 05: Day Boundary → Asia/Riyadh Summary

**One-liner:** Refactored `computeCalendarDay` to anchor day boundaries to `Asia/Riyadh` via `Intl.DateTimeFormat("en-CA", ...)`, closing the UTC-drift bug where users activating at 23:00 Riyadh the night before would incorrectly see the same day number at 06:00 the next morning.

---

## What changed

### `src/lib/calendarDay.ts`

**Before:**
```typescript
const start = new Date(subscriptionStartDate);
start.setHours(0, 0, 0, 0);          // ← anchors to server local TZ (UTC on Vercel)

const today = new Date(now);
today.setHours(0, 0, 0, 0);          // ← anchors to server local TZ (UTC on Vercel)

const diffMs = today.getTime() - start.getTime();
const diffDays = Math.floor(diffMs / MS_PER_DAY) + 1;
```

**After:**
```typescript
function toRiyadhDateString(date: Date): string {
  const fmt = new Intl.DateTimeFormat("en-CA", {
    timeZone: "Asia/Riyadh",
    year: "numeric",
    month: "2-digit",
    day: "2-digit",
  });
  return fmt.format(date); // "YYYY-MM-DD"
}

function riyadhDaysBetween(startIso: string, nowDate: Date): number {
  const startDate = new Date(startIso);
  if (isNaN(startDate.getTime())) return 0;
  const startStr = toRiyadhDateString(startDate);
  const nowStr   = toRiyadhDateString(nowDate);
  const startMidnight = new Date(`${startStr}T00:00:00Z`).getTime();
  const nowMidnight   = new Date(`${nowStr}T00:00:00Z`).getTime();
  return Math.floor((nowMidnight - startMidnight) / MS_PER_DAY);
}
```

- Added file-header JSDoc documenting the invariant, citing RETURN-06 and PITFALLS.md #3.
- Exported signature (`computeCalendarDay(subscriptionStartDate, now?)`) unchanged.
- `TOTAL_DAYS = 28` clamp preserved.
- `null | undefined` graceful handling preserved.
- Added graceful handling of unparseable date strings (returns `0` delta → day `1`).

## RETURN-06 verification vector

ROADMAP Phase 7 Success Criterion #5:
> "A user activated at 23:00 Asia/Riyadh on day 27 who returns at 06:00 the next morning correctly sees Day 28 (not Day 27 stuck due to UTC drift)."

Encoded as test:

```typescript
const subscriptionStartDate = "2026-03-01T20:00:00Z"; // 2026-03-01 23:00 Riyadh (day 1)
const now = new Date("2026-03-29T03:00:00Z");         // 2026-03-29 06:00 Riyadh
expect(computeCalendarDay(subscriptionStartDate, now)).toBe(28); // ✓ passes
```

Manual trace:
- `toRiyadhDateString("2026-03-01T20:00:00Z")` → `"2026-03-01"` (Riyadh is UTC+3, so 20:00 UTC = 23:00 same day)
- `toRiyadhDateString("2026-03-29T03:00:00Z")` → `"2026-03-29"` (03:00 UTC = 06:00 Riyadh same day)
- Midnight-to-midnight delta: 28 calendar days
- `+1` enrollment offset → day 29 → clamped to `28`. ✓

Under the OLD UTC-anchored implementation:
- `setHours(0,0,0,0)` in UTC: both dates floor to the respective UTC midnights
- 2026-03-01 floor → 2026-03-01T00:00Z; 2026-03-29 floor → 2026-03-29T00:00Z
- Delta: 28 days + 1 = 29 clamped → 28 ... so the old impl *happens* to return 28 for THIS exact instant, but the bug surfaces at the edge: e.g., a subscription starting 2026-03-01 20:00 UTC (23:00 Riyadh day 1) evaluated at 2026-03-01 22:00 UTC (01:00 Riyadh day 2) — the old code returns day 1 (same UTC date) while Riyadh has already advanced. The new test `RETURN-06: midnight Riyadh boundary` asserts this scenario returns `2`, which the old code would have failed.
