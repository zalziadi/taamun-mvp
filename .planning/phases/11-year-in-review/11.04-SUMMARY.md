# Phase 11 Plan 04: /year-in-review Server Component Summary

**Ships the `/year-in-review` route: authenticated-only Server Component that calls `getYearInReview(userId)` (Plan 11.03, parallel wave), shows a gentle Arabic nudge for users with <30 reflections (no event, no archive), and fires exactly one `year_review_opened {year_key, reflections_count}` event per successful render before mounting the `YearInReviewArchive` placeholder component that Plan 11.05 will enrich in Wave 3.**

## Accomplishments

- **`/year-in-review` route live** — authenticated-only Server Component, redirects unauth users to `/auth?next=/year-in-review` via `requireUser` + `next/navigation.redirect`.
- **Threshold gate** — when `getYearInReview(userId)` returns `null` (<30 reflections), page renders a two-line Arabic nudge ("حين تتراكم أيامك..." / "ارجع لاحقاً — الوقت ينضج ما بداخله") with **no share button, no event emission, no archive component** — directly enforces YIR-06 ("gentle below-threshold experience") and the "no shaming" tenet of YIR-09.
- **Single-emission analytics** — above-threshold renders call `void emitEvent({name:"year_review_opened", properties:{year_key, reflections_count}}, user.id)` exactly once. `force-dynamic` prevents static-cache double-emission; `void` makes the call fire-and-forget per the Plan 06 emitEvent contract (never throws, never blocks render).
- **Privacy preserved** — the page and placeholder archive import ONLY `YIRPublicStats` (never `YIRPrivateContent`). Event properties are `year_key` (string matching `/^[0-9]{4}_anniversary$/`) and `reflections_count` (number) — both in the `ALLOWED_PROPERTY_KEYS` whitelist from Phase 6.
- **Eastern Arabic numerals** — placeholder archive formats `reflections_count` via `Intl.NumberFormat("ar-SA-u-nu-arab")` (YIR-05). Gate message uses Arabic numeral literal `٣٠` directly.
- **Private-route metadata** — `layout.tsx` exports `robots: { index: false, follow: false }` so crawlers never index this per-user retrospective page.
- **Parallel-wave decoupling** — `YearInReviewArchive` shipped as a minimal placeholder so Plan 11.05 (Wave 3) enriches an existing file rather than creating one, making the Wave 2/3 handoff cleaner.

## Decisions Made

- **requireUser returns a discriminated union, not a redirect-throw.** Reading `src/lib/authz.ts` confirmed `requireUser` returns `{ok: true, user} | {ok: false, response}` where `response` is a JSON 401 meant for Route Handlers. For Server Components, follow the `/account/referral/page.tsx` pattern: `if (!auth.ok) redirect("/auth?next=...")`. Using `auth.response` directly would not redirect a Server Component.
- **yearKey derived inline.** Plan 11.04's action block references `yearKeyForUser` from `@/lib/yearInReview/yearKey` but that helper is owned by Plan 11.03 (parallel Wave 2). To keep 11.04 independently buildable once 11.03 lands, the year key is computed inline via `new Date(activation_started_at || created_at).getUTCFullYear() + "_anniversary"`. When Plan 11.03 ships the shared helper, a later refactor can swap this inline function for the import — the output format is identical and matches `YEAR_KEY_PATTERN`.
- **YearInReviewArchive placeholder shipped now.** Two reasons: (a) without it, `page.tsx` won't type-check even when 11.03 lands; (b) Plan 11.05's enrichment becomes a content edit instead of a new file, which is easier to review and keeps the file-ownership boundary at Wave 2.
- **`void emitEvent` not `await emitEvent`.** The orchestrator brief allows either (emitEvent never throws), but `void` makes the fire-and-forget intent explicit and avoids any theoretical micro-stall if the PostHog fetch takes a moment on cold Node.js runtime. Render is instant; event emission is best-effort.
- **`runtime="nodejs"`.** Explicit opt-out of Edge runtime because `getYearInReview` (Plan 11.03) will transitively import `supabaseAdmin` which uses Node crypto, and `emitEvent` uses `fetch` which works everywhere but runs more predictably on Node per STACK.md.

## Deviations from Plan

- **[Rule 3 – Parallel-wave decoupling] yearKey derived inline instead of importing `yearKeyForUser` from `@/lib/yearInReview/yearKey`.** The plan's action block assumed Plan 11.03 would land first, but both plans run in parallel Wave 2 per the orchestrator brief. Creating that helper here would overlap with 11.03's file ownership. Resolution: inline a minimal `yearKeyFromUser` function in `page.tsx` with identical output format (`YYYY_anniversary`). Matches `YEAR_KEY_PATTERN` in Plan 11.02's types.ts. Files modified: `src/app/year-in-review/page.tsx`. Not a functional change — same output, different origin.
