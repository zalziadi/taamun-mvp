## Accomplishments

- `src/lib/yearInReview/yearKey.ts` — pure-logic helpers; `yearKeyForUser(profile, now?)` picks activation_started_at ?? created_at, maps (month, day) to Asia/Riyadh calendar time, and returns the most-recent-anniversary year. `parseYearKey(key, anchor)` returns `[start, end)` UTC window.
- `src/lib/yearInReview/aggregate.ts` — `getYearInReview(userId, supabase?, now?)`:
  1. Reads profile → computes year_key
  2. Counts reflections in window — < 30 → returns null
  3. Reads year_reviews cache — <24h + valid shape → serves cached
  4. Else calls `supabase.rpc("get_year_in_review", { p_user_id, p_year_key })`
  5. Upserts fresh row with `onConflict: 'user_id,year_key'`
  6. Returns the validated payload
  7. Any throw, any RPC error, any malformed payload → logs + returns null (never throws)

## Decisions Made

- **Return null below threshold (not throw):** Plan's `<interfaces>` block specified `Promise<YIRPublicStats | null>`. Null is the signal for Plan 11.04's page route to render the "ارجع لاحقاً" gate gracefully. Throwing would require a try/catch in every caller — worse UX, more fragile.
- **Threshold gate always hits DB:** Even when a cached row exists, we count reflections live. Reason: the 30-threshold is the YIR-01 user-facing gate; a user who drops below 30 (e.g., data correction) must see the gate again. Cache serves the *payload*, not the *eligibility*.
- **parseYearKey window uses UTC:** Against `created_at` timestamps stored in UTC, a 3-hour Riyadh offset is immaterial at year-window granularity. Simpler code, no DST traps.
- **Soft-warn on upsert failure:** If the upsert errors (disk full, RLS violation, whatever), we still return the freshly-computed payload so the user sees a correct page. Next request regenerates. Prevents a DB write failure from becoming a 500.
- **Added scenario G (missing profile):** Plan specified 6 scenarios; scenario G covers user-not-found defensively. 7 scenarios + 2 constants = 9 aggregate tests total, exceeding the plan's ≥6-tests critical rule.
