# Phase 8 Plan 04: Retroactive Badge Backfill Migration Summary

Pure-SQL retroactive backfill migration seeds historical milestone (day_1/3/7/14/21/28) and cycle_complete badges for every existing user from `reflections.created_at` and `progress.completed_cycles`, with `notified=true` on every row — delivering BADGE-07 and structurally mitigating PITFALL #4 (retroactive event-spam on deploy).

## Objective Achieved

BADGE-07 delivered via a single idempotent SQL migration file:
`supabase/migrations/20260420000000_v1_2_badge_backfill.sql`.

On apply, the day-9 customer (قلبي يتشرب معاني) will see day_1, day_3, day_7 badges already unlocked on her next `/progress` visit with:
- Zero push notifications
- Zero email pushes
- Zero PostHog `badge_unlock` events
- Zero toast / modal / animation interruption

## Implementation

### Migration file structure

Five `INSERT INTO public.badges ... SELECT ... ON CONFLICT DO NOTHING` statements:

| # | Section | Source | Cycle attribution | unlocked_at source |
|---|---------|--------|-------------------|--------------------|
| 1 | Milestones (day_1/3/7/14/21) for cycle 1 | `reflections` WHERE day IN (1,3,7,14,21) | 1 | `MIN(reflections.created_at)` |
| 2 | day_28 for cycle 1 | `reflections` WHERE day=28 | 1 | `MIN(reflections.created_at)` |
| 3 | cycle_complete for archived cycles | `progress.completed_cycles` (unnest) | from array element | `COALESCE(progress.updated_at, now())` |
| 4 | day_28 for archived cycles | `progress.completed_cycles` (unnest) | from array element | `COALESCE(progress.updated_at, now())` |
| 5 | Milestones (day_1/3/7/14/21) for archived cycles | `progress.completed_cycles` × `ARRAY[1,3,7,14,21]` (LATERAL unnest × 2) | from array element | `COALESCE(progress.updated_at, now())` |

Every row sets `notified = true`. Every statement ends in `ON CONFLICT (user_id, badge_code, cycle_number) DO NOTHING`.

Migration timestamp `20260420000000` is one day after Phase 7's migration — no collision in existing migrations directory.

### PITFALL #4 structural mitigation

The file contains NONE of the following (verified by grep):
- `CREATE TRIGGER` / `CREATE FUNCTION` (only in a comment describing what the migration does NOT do)
- `pg_notify` / `NOTIFY` channel
- `pg_net` / `http_*` function calls
- `ALTER TABLE` / `DROP TABLE` / `TRUNCATE`
- Any application code path (it's a `.sql` file, not `.ts`)

Application-level `emitEvent` calls are structurally impossible from raw SQL. Future `unlockBadge(user, code, cycle)` calls on backfilled tuples return 0 inserts due to the UNIQUE constraint + helper's `ignoreDuplicates + length>0` gate → no `emitEvent` fires.

## Known-edge Ziad-flag

**Section 5 over-backfill potential:** A user with `completed_cycles = [1, 2]` archived via an edge case (e.g., manual admin adjustment, or a data-import from a legacy path predating the Phase 7 start-cycle archival gate) who did not actually reach day 21 of cycle 2 will still receive a `day_21` badge for cycle 2 from section 5. This is the documented tradeoff — the ON CONFLICT DO NOTHING means a corrective migration with finer logic can run later without duplication. In practice, cycle archival only ever happens via the start-cycle route which gates on `completed_days.includes(28)`, so this case is vanishingly rare in real data.
