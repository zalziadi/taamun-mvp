   ```typescript
   // --- BADGE-05/06/09: milestone badge fire-and-forget ---
   // Guard on PROGRESSION_MILESTONES (1,3,7,14,21,28) — day_28 is already
   // handled by start-cycle (Phase 7 Plan 07.04) but saving a day_28 reflection
   // here ALSO produces the correct badge idempotently via UNIQUE constraint.
   // Day 28 in this path is harmless: duplicate upsert returns already_unlocked,
   // no duplicate event (PITFALL #5 handled by the helper's emit-only-on-insert gate).
   if ((PROGRESSION_MILESTONES as readonly number[]).includes(day)) {
     const { data: cycleRow } = await supabase
       .from("progress")
       .select("current_cycle")
       .eq("user_id", user.id)
       .maybeSingle();
     const cycle = typeof cycleRow?.current_cycle === "number" ? cycleRow.current_cycle : 1;
     const code = `day_${day}` as const; // narrowed by the guard — TS happy once BadgeCode is widened
     // eslint-disable-next-line @typescript-eslint/no-floating-promises
     void unlockBadge(user.id, code as "day_1"|"day_3"|"day_7"|"day_14"|"day_21"|"day_28", cycle, day);
   }
   ```

4. **No UI side-effect**. Do NOT:
   - Return the unlock result in the response payload (BADGE-09: no modal trigger on client).
   - Fire a push notification here (NFR-04 + 08-CONTEXT banned anti-patterns — unlock is silent).
   - Add a `track()` call for the unlock (the helper already emits `badge_unlock` server-side — never duplicate it).
   - `await` the unlock (fire-and-forget prevents slow DB from affecting reflection-save latency).

5. **Idempotency + safety**:
   - `void` prefix ensures no unhandled-promise linting (per the existing `enhanceReflectionWithAI` pattern in the same file — follow the same style).
   - If the helper returns `{ unlocked: false, reason: "db_error" }`, the error is already logged via `console.warn` inside the helper — no additional handling needed here.
   - The UNIQUE constraint makes duplicate saves idempotent. A day-7 reflection saved twice produces ONE badge row, ONE `badge_unlock` event (the helper's emit-only-on-fresh-insert gate handles it).

6. **Cycle discovery failure**: if the progress table read fails, default cycle to `1` and still fire the unlock. Rationale: on cycle-1 users with a transient DB hiccup, awarding cycle 1 is correct; on cycle-2+ users with a hiccup, the UNIQUE constraint would correctly reject the duplicate on next save because cycle-2's badge already exists separately. Worst case = one misaligned row on a transient error, acceptable given the backfill migration (Plan 08.04) reconstructs correct cycles from reflections.
