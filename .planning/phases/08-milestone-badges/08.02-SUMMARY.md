# Phase 08 Plan 02: Milestone Badge Unlock Trigger — Summary

**One-liner:** Widened `unlockBadge` helper to accept the 7-code `BadgeCode` union + wired a fire-and-forget `void unlockBadge(...)` call into `/api/reflections` POST success path so days 1/3/7/14/21 unlock silently on reflection save — zero client-side code touched, zero schema change, zero new dep.

## What was built

### Final widened `BadgeCode` union shape

```ts
// src/lib/badges/unlock.ts
export type BadgeCode =
  | "day_1"
  | "day_3"
  | "day_7"
  | "day_14"
  | "day_21"
  | "day_28"
  | "cycle_complete";

/** @deprecated Phase 7 alias kept for one commit. Prefer `BadgeCode`. */
export type Phase7BadgeCode = "day_28";
```

Helper signature now accepts the full union:

```ts
export async function unlockBadge(
  userId: string,
  badge_code: BadgeCode,
  cycle_number: number,
  day_number: number,
): Promise<UnlockResult>;
```

Body of `unlockBadge` is unchanged from Phase 7 — the `.upsert(..., { onConflict: "user_id,badge_code,cycle_number", ignoreDuplicates: true }).select()` pattern is data-driven and idempotent across all 7 codes.

### Final `ALLOWED_BADGE_CODES` set (with rationale)

```ts
// src/app/api/badges/unlock/route.ts
const ALLOWED_BADGE_CODES = new Set<BadgeCode>([
  "day_1",
  "day_3",
  "day_7",
  "day_14",
  "day_21",
  "day_28",
]);
```

**`cycle_complete` intentionally excluded** — that badge is awarded server-internally from `/api/program/start-cycle` (Plan 08.03) as part of the cycle-transition success path. Exposing it as a client-triggerable code would allow a client to request an unlock for the cycle-completion badge outside the legitimate transition flow. The server-internal path already holds the `finishedCycle` value and can fire the unlock atomically.

400 response body also renamed: `badge_code_not_in_phase_7_scope` → `badge_code_not_allowed`.

### Call-site location in `/api/reflections` POST

- **File:** `src/app/api/reflections/route.ts`
- **Import line:** 12 (`import { unlockBadge } from "@/lib/badges/unlock";`)
- **Import line:** 13 (`import { PROGRESSION_MILESTONES } from "@/lib/taamun-content";`)
- **Call site line:** **125** (`void unlockBadge(user.id, code, cycle, day);`)

Placed AFTER the reflection upsert success (around the original line ~93) and BEFORE the existing `void enhanceReflectionWithAI(...)` block — guarded by `(PROGRESSION_MILESTONES as readonly number[]).includes(day)`. Current cycle is read from `progress.current_cycle` via a cheap `maybeSingle()` query and defaults to `1` on null/transient-error.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking build error] Unknown ESLint rule in eslint-disable comment**
- **Found during:** Task 2 build verification
- **Issue:** Plan prescribed `// eslint-disable-next-line @typescript-eslint/no-floating-promises` above the `void unlockBadge(...)` call. This project's ESLint config does not register the `@typescript-eslint/no-floating-promises` rule, so the disable comment itself produced a build-breaking error: `Definition for rule '@typescript-eslint/no-floating-promises' was not found`.
- **Fix:** Removed the disable comment. The `void` prefix alone is sufficient — the existing `void enhanceReflectionWithAI(...)` call in the same file (line 112) follows the same pattern without an eslint-disable comment, and passes the existing lint config.
- **Files modified:** `src/app/api/reflections/route.ts`
- **Commit:** `82c6967` (fix inline before commit)

### Vercel-plugin observability suggestion — NOT acted on (inherited from Phase 7)

The Vercel plugin's `posttooluse-validate` hook fired SUGGESTION-level warnings on every Edit of the three files, recommending additional observability logging. Not acted on for the same reasons Plan 07.04 documented:
