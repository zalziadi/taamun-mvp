Critical shape question for the executor to VERIFY BEFORE writing the migration:
- Does `reflections` have a `cycle_number` column? **Probably not** — reflections table predates the v1.2 cycles (v1.0 schema). The backfill therefore MUST derive the cycle for each historical reflection from the user's `progress.current_cycle` + `progress.completed_cycles`. Safest assumption: historical reflections belong to cycle 1 (users who have archived cycles had those archived AFTER their reflections were written).
- Actual algorithm recommendation: for each user, the milestone badges backfill maps to **cycle 1 only** because reflections are keyed on (user_id, day) without a cycle dimension. If a user has advanced to cycle 2/3, cycle_complete badges for those archived cycles come from `progress.completed_cycles`.

Canonical milestone list (match Plan 08.02 / 08.03): `ARRAY[1, 3, 7, 14, 21]` for day_N codes. `day_28` is the Phase 7 cycle-boundary badge — skip it here (ON CONFLICT DO NOTHING protects against duplicates anyway, but explicitly excluding it avoids noise).

**Write the migration file** at `supabase/migrations/20260420000000_v1_2_badge_backfill.sql`:

```sql
-- ============================================================================
-- Phase 8 — v1.2 Retention Loop: Retroactive badge backfill
-- المرحلة 8 — v1.2 حلقة البقاء: تعبئة الشارات الرجعية
-- ============================================================================
-- REQs addressed:
--   * BADGE-07 — existing users are backfilled with historical milestone
--     badges on deploy. Every backfilled row has `notified = true` so no
--     notification/animation/toast fires on first /progress visit post-deploy.
--
-- NFRs upheld:
--   * NFR-08: Pure SQL. No runtime dependencies.
--   * NFR-09: Additive-only. No destructive DDL. No ALTER TABLE. Re-running
--     this migration is a no-op (ON CONFLICT DO NOTHING on every INSERT).
--   * NFR-10: Idempotent — safe to re-apply on any environment.
--
-- Load-bearing invariant (PITFALL #4 — retroactive badge event-spam on deploy):
--   This migration MUST NOT invoke any application code path. No triggers
--   that call pg_net / HTTP endpoints. No NOTIFY channels consumed by a
--   PostHog forwarder. No webhook. The only side effect is INSERT INTO
--   public.badges with notified=true. PostHog events fire from the Phase 7
--   `unlockBadge` helper's `emitEvent` call — a raw SQL INSERT bypasses the
--   helper entirely, which is exactly what BADGE-07 demands.
--
-- Cycle attribution:
--   Reflections do NOT have a cycle_number column (schema predates v1.2
--   cycles). Historical milestone reflections are attributed to cycle 1 —
--   the cycle they were written in. Archived cycles (progress.completed_cycles)
--   each produce a cycle_complete badge keyed on the cycle_number in the array.
-- ============================================================================

-- ----------------------------------------------------------------------------
-- 1. Milestone badges (day_1, day_3, day_7, day_14, day_21) for cycle 1
-- ----------------------------------------------------------------------------
-- For every (user_id, day) tuple in reflections where day ∈ {1,3,7,14,21},
-- insert a badge row with unlocked_at = MIN(created_at) for that tuple.
-- Cycle is always 1 (historical reflections lived under cycle 1 only).
-- notified=true so future unlockBadge() calls on (user,code,1) are no-ops.
-- ON CONFLICT DO NOTHING — safe to re-run.
-- ----------------------------------------------------------------------------

INSERT INTO public.badges (user_id, badge_code, cycle_number, unlocked_at, notified)
SELECT
  r.user_id,
  'day_' || r.day::text AS badge_code,
  1 AS cycle_number,
  MIN(r.created_at) AS unlocked_at,
  true AS notified
FROM public.reflections r
WHERE r.day IN (1, 3, 7, 14, 21)
GROUP BY r.user_id, r.day
ON CONFLICT (user_id, badge_code, cycle_number) DO NOTHING;

-- ----------------------------------------------------------------------------
-- 2. Day-28 milestone badges — cycle 1 only, from reflections
-- ----------------------------------------------------------------------------
-- Phase 7's day_28 unlock fires on cycle TRANSITION (start-cycle route).
-- A user who wrote a day-28 reflection but never tapped "واصل الرحلة" has
-- NO day_28 badge. Backfill it here. ON CONFLICT protects against the Phase 7
-- rows (transitions that already awarded the badge).
-- ----------------------------------------------------------------------------

INSERT INTO public.badges (user_id, badge_code, cycle_number, unlocked_at, notified)
SELECT
  r.user_id,
  'day_28' AS badge_code,
  1 AS cycle_number,
  MIN(r.created_at) AS unlocked_at,
  true AS notified
FROM public.reflections r
WHERE r.day = 28
GROUP BY r.user_id
ON CONFLICT (user_id, badge_code, cycle_number) DO NOTHING;

-- ----------------------------------------------------------------------------
-- 3. cycle_complete badges for every archived cycle in progress.completed_cycles
-- ----------------------------------------------------------------------------
-- For every user with a non-empty completed_cycles array, insert one
-- cycle_complete row per archived cycle_number. unlocked_at defaults to
-- progress.updated_at (best-available timestamp — completed_cycles doesn't
-- carry per-cycle archive times). Falls back to now() if updated_at is null.
-- ----------------------------------------------------------------------------

INSERT INTO public.badges (user_id, badge_code, cycle_number, unlocked_at, notified)
SELECT
  p.user_id,
  'cycle_complete' AS badge_code,
  c.cycle_n AS cycle_number,
  COALESCE(p.updated_at, now()) AS unlocked_at,
  true AS notified
FROM public.progress p,
     LATERAL unnest(COALESCE(p.completed_cycles, ARRAY[]::int[])) AS c(cycle_n)
WHERE c.cycle_n BETWEEN 1 AND 99
ON CONFLICT (user_id, badge_code, cycle_number) DO NOTHING;

-- ----------------------------------------------------------------------------
-- 4. Day-28 milestone badges for ARCHIVED cycles (cycle_number > 1)
-- ----------------------------------------------------------------------------
-- If a user finished cycle 1 (completed_cycles contains 1), Phase 7 awarded
-- day_28 for cycle 1 on the start-cycle transition. But historical cycle-2
-- completions predating Phase 7 deploy would have no day_28 row. Backfill
-- defensively — ON CONFLICT protects Phase 7's rows.
-- ----------------------------------------------------------------------------

INSERT INTO public.badges (user_id, badge_code, cycle_number, unlocked_at, notified)
SELECT
  p.user_id,
  'day_28' AS badge_code,
  c.cycle_n AS cycle_number,
  COALESCE(p.updated_at, now()) AS unlocked_at,
  true AS notified
FROM public.progress p,
     LATERAL unnest(COALESCE(p.completed_cycles, ARRAY[]::int[])) AS c(cycle_n)
WHERE c.cycle_n BETWEEN 1 AND 99
ON CONFLICT (user_id, badge_code, cycle_number) DO NOTHING;

-- ----------------------------------------------------------------------------
-- 5. Milestone badges (day_1..day_21) for ARCHIVED cycles (cycle_number > 1)
-- ----------------------------------------------------------------------------
-- Mirror of section 1 for users who have multi-cycle progress. Since
-- reflections table is keyed on (user_id, day) without cycle, we assume
-- each archived cycle had a full milestone sweep (the user DID reach day 28
-- to archive the cycle). Creates one row per (user, milestone, archived_cycle).
-- unlocked_at uses progress.updated_at as the closest-available timestamp.
-- ----------------------------------------------------------------------------

INSERT INTO public.badges (user_id, badge_code, cycle_number, unlocked_at, notified)
SELECT
  p.user_id,
  'day_' || m.day_n::text AS badge_code,
  c.cycle_n AS cycle_number,
  COALESCE(p.updated_at, now()) AS unlocked_at,
  true AS notified
FROM public.progress p,
     LATERAL unnest(COALESCE(p.completed_cycles, ARRAY[]::int[])) AS c(cycle_n),
     LATERAL unnest(ARRAY[1, 3, 7, 14, 21]) AS m(day_n)
WHERE c.cycle_n BETWEEN 1 AND 99
ON CONFLICT (user_id, badge_code, cycle_number) DO NOTHING;

-- ============================================================================
-- Post-conditions (verification queries — run manually to confirm):
--
-- -- Count backfilled rows:
-- SELECT badge_code, cycle_number, COUNT(*)
-- FROM public.badges
-- WHERE notified = true
-- GROUP BY badge_code, cycle_number
-- ORDER BY cycle_number, badge_code;
--
-- -- Spot-check day-9 customer scenario — user with day_number up to 9 should
-- -- now have day_1, day_3, day_7 badges in cycle 1:
-- SELECT u.id, b.badge_code, b.unlocked_at, b.notified
-- FROM auth.users u
-- JOIN public.badges b ON b.user_id = u.id
-- WHERE b.cycle_number = 1
-- ORDER BY u.id, b.badge_code;
-- ============================================================================

-- ============================================================================
-- DOWN (manual rollback — Supabase migrations are UP-only; copy-paste to revert):
--   -- Removes only backfilled rows; Phase 7 rows (notified=true too, but
--   -- unlocked_at precise) would also be removed. If you need to distinguish,
--   -- filter by unlocked_at matching reflections.created_at vs progress.updated_at.
--   DELETE FROM public.badges WHERE notified = true AND badge_code IN
--     ('day_1','day_3','day_7','day_14','day_21','cycle_complete');
-- ============================================================================
```

<risks_and_mitigations>
- **PITFALL #4 (retroactive badge event-spam on deploy — LOAD-BEARING)**: Mitigated structurally. Migration is pure SQL, cannot invoke `emitEvent`, and `notified=true` ensures future `unlockBadge` calls on the same tuple are no-ops (the helper's `ignoreDuplicates + .select()` returns 0 inserts, gate blocks emitEvent).
- **Reflections table doesn't have `cycle_number`**: Mitigated by attributing all historical reflections to cycle 1 (correct for >99% of users; users with archived cycles have `progress.completed_cycles` which drives sections 3-5).
- **`progress.updated_at` is imprecise for per-cycle timestamps**: Accepted. Sections 3-5 use `COALESCE(progress.updated_at, now())` — worse than reflections.created_at but the only signal we have. Users whose cycle-2 badges all share the same timestamp won't notice (the /progress page shows only the date, not the moment).
- **Migration runs twice (dev + prod, or re-apply)**: Mitigated by `ON CONFLICT DO NOTHING` on every INSERT. Re-applying is a no-op.
- **Race between this migration and a live user completing day 7 at deploy moment**: The live `unlockBadge` helper call would `.upsert(..., ignoreDuplicates: true)`. If the helper runs first, its row is inserted with `notified=true` (Phase 7 default) and MIN(created_at) unlocked_at (helper uses `now()` → fine). If the migration runs first, helper finds the row and skips. Either order is correct; no duplicate, no extra event.
- **Users with day reflections beyond 28 (cycle 2/3 partially completed)**: Sections 5 and 3 give them cycle_complete for archived cycles but CANNOT give them mid-cycle milestones for in-progress cycles (reflections table has no cycle column). Acceptable tradeoff — they'll earn mid-cycle-2 milestones going forward via Plan 08.02's fire-and-forget on next reflection save. Plan 08.06 UAT can probe this for users currently on cycle 2 day 10.
- **Over-backfill risk for sections 4-5**: If a user has `completed_cycles = [1, 2]` but never reached day 21 of cycle 2 (archived early via edge case), the SQL assumes they did. Since cycles are archived via the start-cycle route which gates on `completed_days.includes(28)`, this is vanishingly rare; the `ON CONFLICT DO NOTHING` means re-running with finer logic later is safe.
- **Migration timestamp collision**: The proposed filename `20260420000000_v1_2_badge_backfill.sql` is one day after Phase 7's `20260419000000_v1_2_badges_and_cycle_guard.sql`. Verify no other migration has this exact timestamp; if so, bump to `20260420000001_...`. Executor must `ls supabase/migrations/` first.
</risks_and_mitigations>
